- Add a block eruption mechanic. Left clicking inside a block should allow the player to enter a BlockReady state. Subsequently holding and dragging should trigger an eruption of many blocks along the path of the sweep, out from the starting position. 
    - The tricky thing here is controlling the shape of the eruption. I'd like it to feel like a burst of block creation, that slows to a trickle and ends across the path of the sweep. The total number of blocks created should be proportional to the charge time of the move, with a saturating maximum. 
    - So if, for instance, the player sweeps up vertically, it should create a sort of pyramid, with slope that initially decays quickly and then narrows to a point. 
    - The height of the pyramid should be larger if the player swept faster. For this move, it should always start within the block creation boundary, but should be able to end outside of it without a problem. 
    - An implementation note is that it may be annoying to keep the mouse in one position while moving etc. We should say that the move is charging while the mouse is inside blocks, and erupts from the block at which the player sweeps out for the first time. 
    - So there's a starting point, and a sweep path followed by the player, defined in game coordinates. The sweep path will have both positions and velocities. Moving the mouse fast should result in a long and narrow tower of blocks (although still mostly widest at the base), while moving it slow should produce a large, wide, mound. In either case, the number of blocks produced should be similar, although there should be reasonable limits on the breadth and height of the blocks produced. 
    - There should also be a minimum curvature of the path block production follows. The sweep should be treated as a suggestion for something like a bezier curve that follows. Slower arcs should allow for higher curvature. We can treat it like an optimization problem, in which a relatively low parameter piecewise curve attempts to track the mouse under constraints on derivative / second derivative with respect to time. 
    - given the existence of the sweep curve, one approach would be to assign all blocks in proximity a fill-order priority value. say higher is better, and the initial value is the maximum number of blocks generated by the sweep. 
    - this is all a little complicated. maybe a better way is to simulate a system, like a ball, with significant mass and an inital velocity, being pulled by the mouse. as the ball moves it loses mass. the number of blocks generated in an area is proportional to the mass lost in an area. 
    - in this case, the block eruption event is associated with a mass ball, which is pulled along the path traced by the player mouse, with initial velocity in the direction of the player mouse, and a field of values, corresponding to free mass within tiles. as the mass ball loses mass, it pushes it into the local field. field squares hold mass until they reach the block creation threshold (or unless they already have a tile in their spot), at which point they spawn a tile sprout. after this point, future mass is pushed into adjacent tiles equally. this is an iterative process, but would converge in maybe a dozen steps, so not horribly expensive. then if a tile never fills up, we just round down and never fill it. 
    - so to execute this we would need to trace the path of the player mouse, and model a pulling point on the path, attached to the mass ball by a spring. the pulling point would follow the path the player traced with rate proportional to the mouse speed, but would maybe not be at the exact position of the mouse, but instead just trace the path it followed at a different total rate. this will require some experimentation
